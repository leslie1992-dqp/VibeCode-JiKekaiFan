# 总则与触发时机

## 目标

在项目中维护一套 **代码驱动** 的 Wiki：业务知识、数据模型与接口契约以 Markdown + Mermaid 形式存在，并与源码保持同步，避免文档落后于实现。

## 何时必须更新 Wiki

在以下任一情况发生后，执行者应 **生成新段落或修改已有 Wiki 文件**：

1. **数据库**：新增/修改/删除表、字段、索引、约束或迁移脚本。
2. **接口**：新增/修改/删除 Controller 路由、HTTP 方法、请求/响应 DTO、错误码。
3. **业务逻辑**：功能步骤、状态机、分支条件、与中间件（Redis/Kafka 等）交互方式发生变化。
4. **模块边界**：新增业务模块或调整模块职责划分。

若本次提交仅为格式化、重命名内部私有方法且 **不改变对外行为**，可在提交说明写「Wiki：无变更」，但仍需确认索引页是否需要调整链接。

## 与代码提交的对应关系

- 理想粒度：**每个可独立审查的提交（或 PR）** 附带对应的 Wiki 增量。
- 最小要求：合并到主分支前，Wiki 中与本次变更相关的章节已更新。

## 产出物一览

| 产出 | 说明 |
|------|------|

| 功能与 API | `wiki/api/features-and-apis.md` |

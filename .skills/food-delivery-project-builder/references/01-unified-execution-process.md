# 统一执行流程

## 适用范围

本流程适用于本项目的所有实现类请求：脚手架搭建、API 开发、数据模型开发、模块功能开发、业务流程开发。

## 第 1 步：加载基线

1. 阅读 `design.md`。
2. 根据用户请求确认目标模块。
3. 按以下顺序整理任务清单：
   - 数据层变更
   - API 契约变更
   - 服务流程变更
   - 前端集成变更

## 第 2 步：应用固定架构

严格使用 `design.md` 中定义的包结构：

- `controller/{module}`
- `service/{module}`
- `impl/{module}`
- `mapper/{module}`
- `entity/{module}`
- `dto/{module}`
- `vo/{module}`
- `convert/{module}`
- `resources/mapper/{module}`

## 第 3 步：数据优先实现

1. 按表映射实现或更新实体类。
2. 实现 Mapper 接口与 XML（存在复杂 SQL 时）。
3. 涉及表结构变更时，在 migration 脚本中补充索引与约束。
4. 确保枚举和状态码映射明确。

## 第 4 步：API 契约实现

1. 先实现 DTO/VO。
2. 实现 Controller 路由与请求校验。
3. 实现 Service 接口与 Impl 实现层。
4. 确保统一返回结构：
   - `code`
   - `message`
   - `data`
   - `requestId`
   - `timestamp`

## 第 5 步：业务规则实现

以下规则必须始终执行：

- 受保护接口必须做 JWT 鉴权。
- 创建/支付/秒杀类操作必须具备幂等控制。
- 订单与秒杀必须有库存一致性保障。
- 订单与支付必须有明确状态机流转。

## 第 6 步：中间件集成

1. Redis：缓存、购物车热点数据、库存标记、幂等键。
2. Kafka：异步事件与削峰填谷。
3. Lua：秒杀原子校验。

## 第 7 步：验收检查清单

结束任务前必须校验：

1. API 路径/方法/参数与映射规范一致。
2. 实体字段与数据库映射规范一致。
3. 功能行为与模块清单、流程规范一致。
4. 错误码定义一致且可追踪。
5. 不存在跨模块命名冲突。

## 第 8 步：交付格式

输出结果按以下维度分组：

1. 模块维度
2. 数据/API/业务影响
3. 测试与验证建议
